# Environment Transformations

**Status:** Candidate curriculum operators

## Core Claim

Before adding explanation or difficulty, reshape the environment so the target
model can produce clear evidence.

## Operators

- **Isolate:** expose one primitive or variable.
- **Split:** separate coupled responsibilities.
- **Elevate:** move to the broader problem generating several symptoms.
- **Enable:** make a preparatory change that permits the target action.
- **Ground:** replace prose guesses with source, types, tests, runtime, or other
  observable evidence.
- **Rotate:** express the same structure through another representation,
  abstraction level, or coordinate system so a hidden invariant becomes
  visible.

## Selection

Use an operator when incidental complexity prevents a meaningful prediction,
causal diagnosis, or safe attempt.

Examples:

- Extract a minimal reproduction to isolate semantics.
- Split state management from transport behavior.
- Elevate repeated local fixes into one lifecycle question.
- Add observability before debugging.
- Prefactor a knot before asking the learner to reason about it.
- Compare a workflow, diagram, equation, and executable case that encode the
  same relation.

## Boundaries

- Environment shaping must preserve the selected target friction.
- Refactoring can become avoidance when it has no evidence or return condition.
- The operators are candidate primitives; usage must test whether they
  generalize beyond programming.
